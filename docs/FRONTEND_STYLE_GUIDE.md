# 연(緣) 프론트엔드 디자인 및 작업 가이드

이 문서는 Claude, Codex 등 다른 코딩 에이전트가 현재 프론트엔드의 디자인 언어와 개발 방식을 유지하면서 기능을 추가할 수 있도록 작성한 기준 문서다.

## 1. 제품과 사용자 경험

연(緣)은 관심사, 목표, 강점을 기반으로 협업 상대를 찾는 모바일 중심 소셜 매칭 앱이다.

화면은 다음 인상을 유지한다.

- 친근하지만 가볍지 않은 현대적인 한국형 모바일 앱
- 흰색 바탕과 넉넉한 여백을 중심으로 한 깨끗한 구성
- 인디고, 라벤더, 코랄을 포인트로 사용
- 사용자의 선택에 즉각적이고 부드러운 모션 피드백 제공
- 설명보다 직접 선택하고 진행하는 흐름을 우선
- 모바일을 먼저 설계하고 데스크톱에서는 중앙 정렬된 좁은 콘텐츠 폭 유지

## 2. 기술 스택

- React 19 + TypeScript
- Vite 8
- React Router
- TanStack Query: 서버 상태와 API 요청
- Zustand: 토스트와 공용 UI 상태
- React Hook Form: 인증 폼과 입력 검증
- Tailwind CSS 4 + `src/index.css`: 유틸리티와 화면별 스타일
- Lucide React: 인터페이스 아이콘
- Matter.js: 온보딩 강점 버블 물리 효과
- MSW: 개발 및 테스트 API 목업
- Vitest + Testing Library
- Oxlint

새 라이브러리를 추가하기 전에 기존 스택으로 구현 가능한지 먼저 확인한다. 아이콘은 직접 SVG를 만들지 말고 Lucide React를 사용한다. 물리 동작처럼 검증된 엔진이 필요한 기능은 직접 계산하지 말고 기존 라이브러리를 사용한다.

## 3. 핵심 색상

기본 토큰은 `front/src/index.css`의 `@theme`에 정의되어 있다.

| 용도 | 색상 |
| --- | --- |
| 브랜드 코랄 | `#E8927C` |
| 밝은 코랄 | `#F0A98F` |
| 브랜드 인디고 | `#5B6EE1` |
| 밝은 인디고 | `#6B5FD6` |
| 기본 네이비 텍스트 | `#1B1B3A` |
| 기본 표면 | `#FEFCFA` |
| 인트로 라벤더 | `#8289D2` |
| 인트로 핑크 | `#EAA7A5` |

주요 CTA는 아래 계열의 그라데이션을 사용한다.

```css
background: linear-gradient(105deg, #747dcc 0%, #9a7fc2 55%, #bd89b3 100%);
```

규칙:

- 본문 배경은 흰색 또는 매우 옅은 표면색을 사용한다.
- 긴 본문 텍스트는 네이비, 보조 문구는 중성 회색을 사용한다.
- 제목에서 한 구절만 인디고 또는 보라색으로 강조한다.
- 한 화면을 보라색 계열만으로 채우지 않는다. 흰색, 회색, 코랄을 함께 사용한다.
- 오류는 채도가 낮은 레드 계열을 사용하고 브랜드 색상과 혼동하지 않게 한다.

## 4. 타이포그래피

기본 글꼴:

```css
font-family: 'Noto Sans KR', system-ui, 'Segoe UI', Roboto, sans-serif;
```

브랜드명이나 워드마크에만 `Song Myung`, `Noto Serif KR` 계열을 사용할 수 있다.

- 모바일 페이지 제목: `25px`에서 `31px`, `font-weight: 800`, `line-height: 1.3` 전후
- 폼 제목: 약 `28px`, 굵기 `800`
- 본문 및 입력값: `13px`에서 `15px`
- 필드 라벨: `12px`에서 `13px`, 굵기 `700` 전후
- 보조 설명: `11px`에서 `13px`, 회색, `line-height: 1.5` 이상
- 글자 간격은 음수로 줄이지 않는다.
- 컴팩트한 패널 안에서 과도하게 큰 제목을 사용하지 않는다.

사용자 노출 문구는 컴포넌트에 직접 작성하지 말고 `front/src/i18n/ko.ts`에 키를 추가한다. 접근성용 `aria-label`처럼 짧고 문맥적인 텍스트는 예외로 둘 수 있다.

## 5. 레이아웃

모바일 기준:

```css
min-height: 100svh;
width: min(100%, 420px);
margin: 0 auto;
padding-inline: 22px;
```

- 모바일 안전 영역은 `env(safe-area-inset-top)`과 `env(safe-area-inset-bottom)`을 고려한다.
- 인증 및 온보딩 화면의 콘텐츠 최대 폭은 보통 `420px`에서 `440px`다.
- 고정된 모바일 경험은 `100vh`가 아니라 `100svh`를 사용한다.
- 화면 전체 높이에 맞춰야 할 때는 `min-height: 0`, `flex: 1`, `overflow: hidden`을 함께 검토한다.
- 작은 높이 화면은 `@media (max-height: 740px)`에서 간격과 요소 높이를 줄인다.
- 데스크톱에서도 모바일 UI를 무작정 늘리지 말고 중앙에 제한된 폭으로 유지한다.
- 고정 형식 요소는 `aspect-ratio`, `min-height`, 명시적인 그리드 열을 사용해 레이아웃 이동을 방지한다.

## 6. 컴포넌트 스타일

### 입력창

- 흰색 배경, `14px` 전후 모서리
- `#DEDEE7` 계열의 얇은 회색 테두리
- 왼쪽에 보라빛 회색 Lucide 아이콘
- 포커스 시 인디고 테두리와 낮은 투명도의 외곽 링
- 오류 메시지는 입력창 아래 `12px`로 표시
- 비밀번호 입력에는 보기/숨김 아이콘 버튼 제공
- 모바일 키보드에 맞는 `inputMode`, `type`, `autoComplete` 설정

### 버튼

- 주요 CTA는 화면 폭을 채우고 최소 높이 `54px`에서 `58px`
- CTA는 퍼플-코랄 그라데이션과 흰색 굵은 텍스트 사용
- 일반 버튼 모서리는 `8px`에서 `18px`, 인증 CTA만 알약 형태 허용
- 비활성 상태는 투명도와 채도를 낮추며 클릭 커서를 제거
- 호버 가능한 장치에서만 위로 `2px`에서 `4px` 이동하고 그림자를 강화
- 터치 시 `scale(0.98)` 전후의 짧은 눌림 피드백 제공
- 아이콘 버튼에는 반드시 `aria-label`을 지정

### 선택 카드와 원형 선택지

- 미선택: 흰색 배경, 얇은 회색 테두리, 낮은 그림자
- 선택: 라벤더 또는 인디고 배경, 진한 보라색 테두리
- 선택 순간 짧은 팝 애니메이션 제공
- `aria-pressed`로 선택 상태 노출
- 텍스트와 아이콘이 부모 크기를 바꾸지 않도록 안정적인 크기 지정

### 진행바

- 전체 단계 수만큼 동일한 그리드 열 구성
- 완료 및 현재 단계는 인디고 계열 그라데이션
- 미진행 단계는 밝은 회색
- 숫자 상태를 `현재 / 전체` 형식으로 함께 표시
- 접근성 라벨에 현재 단계를 포함

## 7. 애니메이션 원칙

모션은 상태 변화와 진행 방향을 설명해야 한다.

- 일반 등장: `400ms`에서 `550ms`
- 버튼과 입력 상호작용: `160ms`에서 `190ms`
- 화면 전환: `500ms`에서 `800ms`
- 권장 이징: `cubic-bezier(0.16, 1, 0.3, 1)`
- 입력 필드와 목록은 `40ms`에서 `80ms` 간격으로 순차 등장
- 애니메이션이 끝난 뒤 호버 스타일과 충돌하지 않도록 `backwards` fill mode를 우선 검토
- 슬라이드처럼 큰 면을 움직일 때는 `translate3d`를 사용해 합성 레이어에서 처리
- `clip-path` 좌표 자체를 매 프레임 바꾸는 애니메이션은 모바일 끊김을 유발할 수 있으므로 피한다.
- `transform`을 물리 엔진과 CSS 애니메이션이 동시에 제어하지 않게 한다.
- 모든 주요 모션에는 `prefers-reduced-motion: reduce` 대체 상태를 제공한다.

Matter.js 버블 UI에서는 다음을 지킨다.

- 위치와 충돌은 Matter.js가 전담한다.
- DOM은 텍스트와 접근성, 스타일 렌더링만 담당한다.
- 크기 변경 시 물리 바디도 같은 비율로 변경한다.
- 크기 변경 후 모든 바디의 수면 상태를 해제해 중력이 다시 적용되게 한다.
- 물리 엔진이 지정하는 `transform`을 CSS keyframe으로 덮지 않는다.
- 컴포넌트 해제 시 Runner, Events, ResizeObserver, Engine을 모두 정리한다.

## 8. 반응형 및 접근성

- 모든 화면은 최소 `360px` 폭과 짧은 모바일 높이를 고려한다.
- 텍스트가 버튼이나 카드 밖으로 넘치지 않도록 `overflow-wrap`과 안정적인 크기를 사용한다.
- 링크와 버튼은 키보드 포커스 상태를 제공한다.
- 아이콘만 있는 버튼은 이름을 제공하고 장식 아이콘은 `aria-hidden` 처리한다.
- 선택 UI에는 `aria-pressed`, 오류에는 `role="alert"`를 사용한다.
- 폼 라벨과 입력을 의미적으로 연결한다.
- 색상만으로 상태를 설명하지 않고 테두리, 굵기, 크기 변화를 함께 사용한다.
- 터치 대상은 가급적 `44px` 이상으로 만든다.

## 9. 상태 및 데이터 처리

- 서버 상태는 TanStack Query 훅에서 관리한다.
- API 호출은 `front/src/api/client.ts`를 통한다.
- 공용 UI 상태만 Zustand에 둔다.
- 일시적인 폼 입력과 화면 단계는 컴포넌트 로컬 상태를 우선한다.
- 온보딩처럼 단계 사이에 유지해야 하는 임시 값은 `sessionStorage`를 사용할 수 있다.
- 저장 데이터 파싱은 `try/catch`와 타입 확인으로 방어한다.
- 사용자 피드백은 공용 토스트를 사용한다.
- API 성공 후 Query 캐시를 갱신하고 필요한 화면으로 이동한다.

## 10. 파일 구조와 책임

```text
front/src/
├── api/             # HTTP 클라이언트
├── components/      # 도메인별 재사용 컴포넌트
├── hooks/           # API 및 서버 상태 훅
├── i18n/            # 사용자 노출 문구
├── mocks/           # MSW 핸들러
├── pages/           # 라우트 단위 화면
├── routes/          # 레이아웃과 접근 제어
├── store/           # 공용 UI 상태
└── types/           # API와 도메인 타입
```

- 페이지는 흐름과 상태를 조합한다.
- 복잡한 상호작용은 별도 컴포넌트로 분리한다.
- API 로직을 페이지 JSX에 직접 작성하지 않는다.
- 동일한 사용자 노출 문자열을 여러 컴포넌트에 복제하지 않는다.
- 작은 변경을 위해 새로운 추상화나 전역 상태를 만들지 않는다.

## 11. 에이전트 작업 순서

다른 에이전트는 아래 순서로 작업한다.

1. 요청과 관련된 페이지, 컴포넌트, 훅, 타입, i18n 키를 먼저 읽는다.
2. 기존 색상과 컴포넌트 패턴을 찾아 재사용한다.
3. 모바일 `360px` 전후와 데스크톱 제한 폭에서 레이아웃을 설계한다.
4. 사용자 노출 문구를 `i18n/ko.ts`에 추가한다.
5. UI 상태, API 상태, 공용 상태의 소유 위치를 분리한다.
6. 키보드, 스크린 리더, 터치 상태를 함께 구현한다.
7. 작은 화면 높이와 `prefers-reduced-motion`을 처리한다.
8. 빌드, 테스트, 린트, diff 검사를 실행한다.
9. 생성된 `dist/`는 커밋하지 않는다.
10. 사용자 변경이나 관계없는 파일을 되돌리지 않는다.

필수 검증 명령:

```bash
cd front
npm run build
npm test -- --run
npm run lint
git diff --check
```

## 13. 에이전트용 실행 프롬프트

아래 문장을 Claude 또는 다른 코딩 에이전트의 작업 지침에 포함할 수 있다.

```text
이 저장소의 프론트엔드를 수정할 때 docs/FRONTEND_STYLE_GUIDE.md를 디자인 및 구현 기준으로 사용하라. 먼저 관련 코드를 읽고 기존 React, TypeScript, Tailwind CSS, Lucide, TanStack Query, Zustand, MSW 패턴을 재사용하라. 모바일 360px 폭과 짧은 화면 높이를 우선 지원하고, 데스크톱에서는 콘텐츠 폭을 제한하라. 사용자 문구는 i18n/ko.ts에 추가하고 접근성 상태, reduced motion, 로딩, 오류, 비활성 상태를 구현하라. 관계없는 코드는 수정하지 말고 완료 전에 build, test, lint, git diff --check를 실행하라.
```
